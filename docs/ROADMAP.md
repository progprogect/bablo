# Roadmap — этапы разработки

Каждый этап — самостоятельный, логически завершённый инкремент. Реализуем строго по порядку,
следующий этап начинаем после приёмки предыдущего.

## Этап 0 — Скелет проекта и деплой
**Цель:** пустое, но работающее приложение на Railway, устанавливаемое как PWA на iPhone.

- Монорепо (client + server, npm workspaces), TypeScript everywhere.
- Fastify-сервер: healthcheck, раздача статики клиента.
- React + Vite + Tailwind, PWA-манифест + service worker, нижняя таб-навигация (3 пустых экрана).
- PostgreSQL на Railway, Drizzle + первая миграция.
- Деплой на Railway, проверка установки PWA на iPhone.

**AC:** приложение открывается по URL Railway; ставится на домашний экран iPhone и запускается standalone; healthcheck зелёный.

## Этап 1 — Админка: ключи и активы
**Цель:** подключение к BingX и управление активами.

- Вход по PIN (сессии).
- Экран админки: ввод BingX API key/secret (шифрование в БД), проверка соединения (запрос баланса).
- CRUD активов: символ + плечо (сид: TIA 20x, TAO 25x, VIRTUAL 25x, WLD 20x).
- BingX REST-клиент (подпись запросов, баланс, информация о контрактах).

**AC:** ключи сохраняются и валидируются; на дашборде виден реальный баланс; активы редактируются и отражаются в табах.

## Этап 2 — Открытие сделки (ядро)
**Цель:** полный двухшаговый флоу открытия сделки.

- Дашборд: табы активов, текущая цена (пока REST-запрос по действию, realtime — этап 4).
- Шаг 1: объём (кол-во монет) + цена SL → Купить/Продать → market-ордер
  (изолированная маржа, one-way, плечо актива) + немедленный SL-ордер.
- Шаг 2: TP вручную или пресетом R/R (от фактической цены входа) → выставление на бирже.
- Карточка активной сделки: все параметры + цена ликвидации.
- Блокировка «уже есть активная сделка».

**AC:** сделка открывается на реальном аккаунте BingX за ≤4 действия; SL/TP выставлены на бирже; карточка показывает корректные данные; вторую сделку открыть нельзя.

## Этап 3 — Риск-движок
**Цель:** принудительное соблюдение риск-плана.

- Таблица лестницы уровней (сид из RISK_ENGINE.md), редактируемая в админке.
- Проверка объёма при открытии (по подтверждённой интерпретации) + подсветка поля.
- Дневные счётчики R, все 4 блокировки (включая часовой кулдаун после каждой сделки),
  пояснение причины + таймер разблокировки в UI.
- Учёт результата закрытой сделки в прогрессе уровня.
- Юнит-тесты на чистую логику риск-движка.

**AC:** сделка сверх лимита не отправляется; все 4 блокировки срабатывают и снимаются вовремя; при блокировке видны причина и таймер; прогресс уровня меняется по результатам сделок; тесты зелёные.

## Этап 4 — Realtime без поллинга
**Цель:** живые данные без циклических запросов.

- WS-коннектор BingX: market stream (цены активов) + account stream (listenKey, продление, реконнект).
- Детект закрытия по SL/TP через ACCOUNT_UPDATE + событийная REST-сверка.
- SSE-канал к клиенту, живое обновление цены, баланса, карточки сделки, блокировок.

**AC:** цена и PnL обновляются без перезагрузки; закрытие по стопу на бирже отражается в приложении и запускает риск-логику; в логах нет циклических REST-опросов.

## Этап 5 — История и статистика
**Цель:** экран истории с инсайтами.

- Список сделок со всеми полями (дата, актив, направление, длительность, R/R, результат %).
- Инсайт «время дня прибыльных сделок».
- Пагинация/фильтры по минимуму.

**AC:** все закрытые сделки видны с корректными данными; инсайт считается верно.

## Этап 6 — Геймификация: дерево роста
**Цель:** визуальная мотивация следовать плану.

- Экран/шит с деревом уровней: пройденные, текущий с прогрессом, будущие.
- Компактный индикатор уровня на дашборде.
- Анимация перехода уровня.

**AC:** дерево отражает реальный прогресс risk_state; переход уровня анимируется; дашборд не перегружен.

## Этап 7 — Расширенный трекинг и сигналы
**Цель:** данные для глубокого анализа во вкладке «Статистика».

- Трекер активной сделки: MFE, пересечение безубытка (по тикам WS).
- Определение и фиксация базовых сигналов (набор согласуем отдельно).
- Вкладка «Статистика» в истории с этими данными.

**AC:** по каждой новой сделке зафиксированы MFE и флаг безубытка; данные видны в статистике.

MFE и безубыток реализованы полностью (трекер по WS-ценам, без поллинга) — см.
[RISK_ENGINE.md](./RISK_ENGINE.md#расширенный-трекинг-сделки-для-статистики) за точной
формулировкой. Конкретный набор «сигналов» (пересечения/структура) в исходном плане
явно оставлен открытым («согласуем отдельно») — в БД под них зарезервировано поле
`signals` (jsonb), логика их вычисления добавится, когда набор будет определён.

## Этап 8 — Редизайн Истории: инсайты и месячная статистика
**Цель:** сделать вкладки «Сделки» и «Статистика» полезными для анализа, а не просто журналом.

- Карточки сделок: реализованный результат в USDT вместо %; причина закрытия показана
  только для SL/TP, для «на бирже»/«вручную» подпись не выводится вовсе (15.07.2026 —
  сначала объединили обе метки в «Вручную», затем убрали слово совсем как неинформативное).
- Карточка-подсказка на вкладке «Сделки»: почасовые инсайты (прибыльные/стоп-часы, часы без
  сделок — склеены в диапазоны с раскрытием по кнопке, а не список из 24 строк), самый
  прибыльный актив, типичный час достижения дневной цели +NR (`server/src/history/insights.ts`).
- Вкладка «Статистика» **репурпожена**: вместо построчного MFE/безубыток-трекинга по каждой
  сделке — карточки по месяцам (`server/src/history/monthlyStats.ts`): % к депозиту, сумма R,
  винрейт, TP/SL/безубыток, торговые дни, разбивка по пресетам R/R. Данные MFE/`beCrossed`
  продолжают собираться трекером и хранятся в БД, но сейчас не выводятся в UI — при
  необходимости для них нужно будет спроектировать отдельное место (например, разворачиваемая
  деталь на карточке сделки).
- Новая таблица `equity_snapshots`: один снимок эквити в день, лениво создаётся при загрузке
  дашборда — база для «% к депозиту».

**AC:** обе вкладки истории отражают новый дизайн; тесты на инсайты и месячную статистику
зелёные; для месяцев без снимка эквити % корректно отсутствует (не падает и не искажается).

### Добавление: график роста депозита (14.07.2026)

По явному запросу пользователя на вкладку «Статистика» добавлена кнопка «График роста
депозита» → полноэкранный шит с кривой линией по `equity_snapshots` (`GET
/api/stats/equity-history`, `client/src/screens/history/EquityChart.tsx` — ручной SVG,
без библиотек графиков). Это осознанное и единственное исключение из принципа
«никаких графиков» — см. `docs/PROJECT.md#принципы-продукта`.

### Правка: % к депозиту доступен для всей истории, не только «вперёд» (15.07.2026)

Изначально % к депозиту работал только «вперёд» от появления `equity_snapshots` (14.07.2026):
для месяцев без снимка ровно на начало месяца % не считался. Пользователь захотел, чтобы
уже существующие сделки тоже участвовали в статистике — переписали расчёт базы на
восстановление «в обратную сторону» от последнего известного снимка (anchor): текущий
баланс минус реализованный PnL всех сделок, закрытых с начала нужного месяца. Так % теперь
доступен для всей истории сделок, а не только для месяцев после появления снимков.

Единственный риск такого восстановления — пополнения/выводы средств между целевым месяцем
и текущим моментом искажали бы результат на их сумму. Добавлена таблица
`equity_adjustments` (date, amountUsd, note) и раздел в админке
(`client/src/screens/admin/EquityAdjustmentsSection.tsx`,
`GET/POST/DELETE /api/admin/equity-adjustments`) — пользователь вручную регистрирует
пополнения/выводы, они учитываются в восстановлении базы наравне с PnL сделок.

### Правка: правило «2 стопа за день» и фикс бага классификации закрытий (16.07.2026)

Добавлено новое дневное ограничение: 2 и более сделки за день, закрытые по стопу (в любом
порядке, независимо от суммы R), блокируют торговлю до сброса дня — как и лимиты −2R/+3R,
но считается отдельно (`daily_stats.sl_count`, `risk/limits.ts#evaluateDailyLimitBlocks`,
`risk/service.ts#recordTradeClose`). Мотивация: даже если убыток компенсирован прибылью по
сумме R, два подряд/за день промаха по стопу — сигнал потери дисциплины входа.

При разборе жалобы пользователя на неверные подсчёты в подсказках («Сделки») и месячной
статистике нашли реальный баг, а не просто проблему форматирования: `getOrderStatus` в
`bingx/client.ts` не распаковывал ответ BingX `{ data: { order: {...} } }` (как и `placeOrder`,
BingX оборачивает результат в `order`), поэтому `status` всегда был `undefined`. Из-за этого
резервная REST-сверка `reconcilePositionFlat` никогда не могла определить FILLED-статус SL/TP
и ВСЕГДА помечала закрытие как `"external"`, даже когда сделка реально закрылась по стопу или
тейку. Именно поэтому все закрытые сделки в истории показывались без подписи «По стопу»/«По
тейку», а TP/SL/БУ в месячной карточке были нулевыми при непустом списке сделок. Баг
исправлен + добавлен тест (`bingx/client.test.ts`). Для уже закрытых «external»-сделок добавлен
восстановительный путь: `POST /admin/reclassify-trades` (`trades/reclassify.ts`) повторно
запрашивает статус сохранённых `bingxOrderIds` и чинит `closeReason`/результат, пока BingX ещё
отдаёт данные по ордеру; кнопка «Пересверить закрытые сделки» в админке
(`client/src/screens/admin/BingxKeysSection.tsx`).

Также уточнена логика инсайтов на вкладке «Сделки» (`history/insights.ts`):
- «Самые прибыльные часы» — раньше час считался прибыльным, если хоть одна сделка в нём была
  прибыльной (`resultR > 0`), и показывалось число прибыльных сделок. Переопределено на долю
  сделок, закрытых именно по тейку: час попадает в список, только если `tpCount / total > 50%`;
  формат подсказки — `9-10ч — 4/5 TP` (открыто 5, из них 4 по тейку).
- «Самый прибыльный актив» — раньше показывалось только число сделок по тейку без знаменателя
  (`(0 сделок по тейку)`, что выглядело как ошибка на малой выборке). Теперь — доля от всех
  сделок по этому активу: `TAO — 1/2 TP (50%)`.
- Формат диапазона часов в подсказках сокращён до `9-10ч` (было `09:00–10:00`) — компактнее и
  единообразно для обеих строк («прибыльные часы» и «часто идут в стоп»).

В карточке месяца на вкладке «Статистика» убрано дублирование слова после числа торговых дней
(«Торговых дней: 2 торговых дней» → «Торговых дней: 2») — лейбл уже описывает значение.
